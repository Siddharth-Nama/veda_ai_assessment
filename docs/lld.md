# Low-Level Design (LLD)

Database Models

I used Mongoose to define the data structure for assignments and their results.

Assignment Schema
Contains the basic info for each task:
- title, subject, and class level.
- dueDate and section.
- questionConfigs follows the user's requirements for question types, counts, and marks.
- status tracks if the AI is still working or if it's done.

Result Schema
Stores the actual question paper generated by the AI:
- Links back to the assignmentId.
- Includes general info like schoolName and time allowed.
- Organized into sections, where each section has an array of questions.
- Each question has text, an answer, a difficulty score, and its marks.

WebSocket Events

The system uses Socket.io to give the user live feedback. 

Key Events:
- join: The client joins a room based on the assignmentId.
- generation:progress: Sends progress percentage and status text to the UI.
- generation:complete: Notifies the client when the paper is ready.
- generation:error: Sends any error messages if the generation fails.

AI Generation logic

The aiService module handles the communication with Gemini. 

Logic Steps:
1. It builds a text prompt using the assignment details.
2. It asks the AI to return only a specific JSON structure.
3. It cleans the AI's response to make sure it's valid JSON.
4. It checks that all sections and questions are present before returning the data to the worker.

UI and Responsiveness

The AppShell handles the layout changes between mobile and desktop. It uses a custom hook to detect screen size and then decides which components to show. It also uses a mounting guard to prevent hydration issues during the initial load in the browser.
